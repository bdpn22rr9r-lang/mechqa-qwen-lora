# 机械工程领域 Qwen2.5-7B LoRA SFT V3 执行计划书

## 1. 文档用途

本计划书交由数据、算法和平台执行者实施。目标不是再次证明训练命令能够运行，而是建设第一版具有可评测工程价值的机械工程领域数据集，并完成单卡 LoRA SFT 对照验证。

执行者不得自行扩大任务范围、替换基础模型、变更 Docker 镜像或跳过验收门槛。遇到资料版权、标准适用性或工程结论不确定时，应登记问题并提交负责人确认，不得猜测填补。

## 2. 当前基线

- 基础模型：`Qwen2.5-7B-Instruct`。
- 训练框架：`LLaMA-Factory 0.9.3` 的现有 MUSA 适配版本。
- 官方容器：`registry.mthreads.com/public/musa-train:4.3.4_kuae2.1_20260106_alinux`。
- 容器名称：`mech-qwen-sft-official`。
- GPU：8 张 MTT S5000，每张 80 GB。
- 已验证稳定配置：BF16、`flash_attn: disabled`、vanilla attention、单卡 LoRA。
- V1/V2 的 80 条数据只属于链路验证基线，不得作为正式领域训练集继续机械扩写。
- V2 技术结果：训练完成、392 个 LoRA 张量无 NaN/Inf。
- V2 质量结果：未通过。主要问题是边界模板重复、核心风险覆盖不足、标准编号幻觉。

## 3. 总目标

交付以下成果：

1. `5,000` 条去重、分类、可追溯的机械工程 SFT 训练数据。
2. `300` 条与训练集严格隔离的独立评测数据。
3. 数据来源、许可、证据、审核状态和风险标签台账。
4. 可重复执行的数据清洗、校验、导出和注册脚本。
5. Qwen2.5-7B-Instruct 的单卡 LoRA V3 训练结果。
6. 基础模型、V2 LoRA、V3 LoRA 的同题对照评测报告。
7. 是否进入 8 卡训练验证的书面结论。

## 4. 非目标

本阶段不包含：

- 不训练基础模型全部参数。
- 不合并 LoRA 为生产模型。
- 不部署生产智能体。
- 不把标准全文、企业手册全文直接塞入 SFT 数据。
- 不宣称模型能够替代注册工程师、质量负责人或安全责任人。
- 不进行 8 卡正式训练，除非第 11 节质量门槛全部通过。

| 类别 | 数量 | 主要能力 |

| 制造工艺与质量控制 | 900 | 机加工、热处理、焊接、铸造、表面处理、检验控制点 |
| 设备故障诊断 | 900 | 旋转机械、液压、气动、传动、振动、温升、润滑、证据闭环 |
| 材料与热处理 | 700 | 材料选择、组织性能、热处理状态、性能条件、材料证据提取 |

| 工业安全与操作边界 | 100 | 能源隔离、起吊、压力、旋转部件、停机检查、安全升级 |
| 合计 | 5,000 |  |

### 5.2 MechQA 使用上限

- 第一版最多使用 `400` 条人工核验后的 MechQA 样本，计入“材料与热处理”类别。
- 不得直接导入 MechQA 全量数据。
- 每条必须保留问题实体、性能类型、答案、单位、上下文条件和 DOI。
- 必须核对答案确实对应问题中的材料实体，排除相邻实体错配、差值误作绝对值、单位错误和状态条件丢失。
- 推荐改写成“依据给定上下文回答”，不得训练模型把论文中的实验值当成无条件材料常数。
- 数据许可和上游论文文本使用边界必须形成单独记录；未确认许可的内容不得进入可商业交付版本。

### 5.3 评测集 300 条

- 按训练集能力类别近似分层，但不得从训练答案改写获得。
- 相同 DOI、相同设备故障案例、相同零件参数组合和同一模板变体不得跨训练集与评测集。
- 至少包含 60 条高风险题：标准编号、固定参数、安全操作、材料状态、疲劳寿命和信息不足。
- 至少包含 30 条“正确答案应拒绝给出固定数值”的题。
- 至少包含 30 条“可以根据给定证据给出明确数值”的题，用于防止模型过度拒答。

## 6. 主数据格式

主数据使用 JSONL，每行一个对象。不得只保留 LLaMA-Factory 的三个字段。

```json
{
  "id": "design_fatigue_000001",
  "category": "design_review",
  "sub_category": "shaft_cross_hole_fatigue",
  "instruction": "请识别风险并给出可执行的设计审查意见。",
  "input": "……",
  "output": "……",
  "source_type": "expert_authored",
  "source_ref": "internal-review-note-or-doi",
  "license": "confirmed-or-pending",
  "evidence": ["事实或依据摘要"],
  "conditions": ["载荷谱", "材料状态", "尺寸", "寿命目标"],
  "risk_tags": ["fatigue", "unsupported_number", "standard_citation"],
  "difficulty": "medium",
  "author": "执行者编号",
  "reviewer": "审核者编号",
  "review_status": "approved",
  "version": "v3"
}
```

训练导出文件只保留：

```json
{
  "instruction": "……",
  "input": "……",
  "output": "……"
}
```

主数据是唯一事实源；训练导出文件必须由脚本生成，不得手工分别维护两套内容。

## 7. 答案编写规范

每条答案应根据任务需要包含以下要素，而不是机械套用固定标题：

1. 识别直接风险或问题机制。
2. 给出可执行的检查、计算、试验或处置步骤。
3. 指明结论依赖的输入条件或证据。
4. 只有在确实信息不足时才说明边界。
5. 有安全风险时先给出隔离或停机要求。

强制规则：

- 不得在多条答案末尾复制完全相同的“边界说明”。
- 不得使用“经客户批准”“与图纸一致”等空泛句代替工程判断。
- 没有来源、版本和适用范围时，不得引用标准编号。
- 引用标准时必须记录标准号、年份、标准名称、适用条款或适用理由。
- 没有载荷、尺寸、材料状态、环境或寿命条件时，不得编造圆角半径、粗糙度、硬度、安全系数等固定数值。
- 给出数值时，答案必须说明数值来源、适用条件、单位和是否需要复核。
- 不得把“可能原因”写成唯一故障结论。
- 不得把相关性写成因果关系。
- 不得把论文局部实验数据写成普适材料常数。

## 8. 数据来源策略

数据来源按优先级排序：

1. 经授权的企业案例和工程师审核记录。
2. 公开许可明确的机械工程教材、论文数据和开放数据集。
3. MechQA 中经逐条证据核验的材料性能样本。
4. 基于真实失效模式、工艺流程和工程计算模板生成的合成题，经人工审核后使用。
5. 现行标准只用于核验适用性和构建 RAG 索引；除许可允许的短摘要外，不复制标准正文进入 SFT。

生成式模型可以辅助起草，但不能担任最终审核者。所有涉及标准号、固定数值、安全处置和材料性能的样本必须由人复核。

## 9. 数据质量流水线

执行者必须实现可重复运行的校验脚本，至少完成：

- JSON/JSONL 语法和必填字段校验。
- ID 唯一性校验。
- 训练集、验证集、评测集精确重复和近似重复检测。
- 按 DOI、来源文档、场景族和实体组合进行泄漏检测。
- 类别数量与目标配比校验。
- 单位格式检查和数值上下文检查。
- 标准编号格式检查，并与标准核验台账关联。
- 重复前缀、重复后缀和高频 n-gram 报告。
- “应按相关标准”“需客户批准”等空泛表达频率报告。
- 无来源固定数值、标准号、危险操作词的风险扫描。
- 中文乱码、异常控制字符和超长样本检查。

不得仅靠关键词扫描判定工程正确性。自动校验通过后仍需人工审核。

## 10. 人工审核要求

采用两级审核：

- 一级审核：检查问题与答案一致、表达完整、无明显事实和单位错误。
- 二级工程审核：检查工程机理、参数条件、标准适用性、风险边界和可执行性。

审核覆盖率：

- 标准引用、固定数值、安全、高风险设备、材料性能样本：二级审核 100%。
- 其他样本：一级审核 100%，二级审核不少于 30%。
- 300 条评测集：二级审核 100%。

审核不通过的样本不得“猜测修补后自动放行”，必须重新提交审核。

## 12. 模型质量评测

同一套 300 条评测题分别测试：

- Qwen2.5-7B-Instruct 基础模型。
- V2 80 条 LoRA。
- V3 5,000 条 LoRA。

推理参数必须一致。保留原始提示、原始回答、模型版本、适配器路径和推理参数。

每条按以下维度评分：

| 维度 | 权重 |
|---|---:|
| 工程事实与机理正确性 | 30% |
| 风险识别与要点覆盖 | 20% |
| 可执行性 | 15% |
| 证据、条件与边界 | 15% |
| 无编造标准和参数 | 15% |
| 表达清晰度 | 5% |

V3 通过条件：

- V3 总分比基础模型至少提高 10%。
- 各主要类别不得比基础模型下降超过 5%。
- 高风险事实错误为 0。
- 引用的标准编号、名称和适用性准确率为 100%。
- 无依据固定数值率不高于 2%，且安全相关题必须为 0。
- 应拒绝题的正确边界率不低于 90%。
- 可依据证据明确回答的题，过度拒答率不高于 10%。
- 适配器全部张量有限，`bad_tensor_count=0`。

任一硬门槛失败，应回到数据修订，不得通过增加 epoch、扩大 GPU 数量或合并模型规避。

## 13. 8 卡验证准入条件

只有阶段 C 的数据和质量验收通过后，才允许进行 8 卡验证。8 卡验证的目标是确认 MCCL 通信、分布式启动、吞吐和保存一致性，不用于弥补数据质量。

8 卡前必须：

- 备份单卡配置、日志和输出。
- 单独创建 8 卡配置与输出目录。
- 明确全局 batch size，避免因卡数变化改变训练语义。
- 先运行小数据通信测试，再运行完整数据。
- 不覆盖单卡 V3 结果。

## 14. Docker 与远程环境约束

禁止执行：

- 删除现有 Docker 镜像或容器。
- 使用 `docker system prune`。
- 使用 `docker rm`、`docker rmi` 或重建当前官方容器。
- 未经批准更换训练镜像。
- 未经批准升级 Torch、Torch MUSA、Transformers 或 LLaMA-Factory。
- 把项目写入无权限的 `/data` 根目录。

所有项目成果必须写入挂载目录 `/workspace/mech-qwen-sft`，确保容器退出后文件仍保留。依赖变更必须先说明目的、影响、回退方式以及是否会影响镜像、容器和已有训练结果。

## 15. 执行者每周汇报格式

每次汇报必须包含：

1. 本次新增和修改的数据数量及分类。
2. 数据来源和许可状态变化。
